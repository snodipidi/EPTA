## Модель токенов

Пара JWT, оба возвращаются **в теле JSON** (не в cookie):

| Токен | TTL (деф.) | Назначение | Где хранит фронт |
|---|---|---|---|
| **access** | `900s` (15 мин) | `Authorization: Bearer` на каждый запрос | localStorage |
| **refresh** | `30d` | обмен на новую пару, когда access протух | localStorage |

`AuthResponseDto` (ответ register / login / refresh):

```json
{
  "accessToken": "eyJ...",
  "refreshToken": "eyJ...",
  "expiresIn": 900,
  "user": { "id": "uuid", "email": "...", "username": "...", "displayName": "...", "role": "USER" }
}
```

**Полезные нагрузки токенов:**

- access (`JwtPayload`): `sub` (user id), `email`, `username`, `role`.
- refresh (`RefreshTokenPayload`): `sub`, `jti` (= `RefreshToken.id`),
  `family` (UUID, объединяет токены одного логина).

## Потоки

### Регистрация (`POST /auth/register`)

1. Пароль хешируется (`PasswordService`, argon2id).
2. В одной транзакции создаются `User` + `Profile` + подписка `FREE`.
3. `TokenService` выдаёт пару токенов (новая `family`).
4. Возвращается `AuthResponseDto`.

### Вход (`POST /auth/login`)

1. Поиск пользователя по email (case-insensitive).
2. Проверка пароля (константное время — защита от timing-атак).
3. Проверка `status === ACTIVE`.
4. Выдача пары токенов. Сохраняются `userAgent` и `ip` сессии.

### Обновление (`POST /auth/refresh`)

Передаётся refresh-токен → `TokenService.rotate()` верифицирует, ротирует и
возвращает **новую пару**. Старый токен отзывается.

### Выход

- `POST /auth/logout` — отзыв одного refresh-токена (текущее устройство).
- `POST /auth/logout-all` (JWT) — отзыв всех активных сессий пользователя.

## Ротация и обнаружение кражи

`TokenService` (`auth/services/token.service.ts`). Refresh-токены хранятся в
таблице `refresh_tokens` **только как SHA-256 хеш** (`tokenHash`) — сырое
значение в БД не попадает.

Алгоритм `rotate(refreshToken)`:

1. Проверить подпись JWT.
2. Найти строку по `jti` из payload.
3. Сверить `tokenHash` с `sha256(предъявленный токен)`.
4. **Reuse detection:** если у строки уже стоит `revokedAt` — токен предъявлен
   повторно (вероятная кража) → **отозвать всю `family`**.
5. Проверить срок (`expiresAt`).
6. В транзакции: создать новую пару в **той же** `family`, пометить старый токен
   `revokedAt` и проставить `replacedById` → id нового (аудит цепочки ротаций).

Так украденный refresh-токен «самоуничтожает» сессию: как только настоящий
владелец или вор обновятся повторно, вся семья отзывается.

## Хранение паролей

`PasswordService` — **argon2id** (memory-hard, рекомендация OWASP):

| Параметр | Значение |
|---|---|
| тип | `argon2id` |
| memory cost | 19 456 KiB (≈ 19 МиБ) |
| time cost | 2 итерации |
| parallelism | 1 |

Смена пароля — `POST /users/me/password` (`currentPassword` + `newPassword`).

## Проверка на каждом запросе

`JwtStrategy` (Passport) не доверяет токену слепо:

1. Достаёт Bearer из заголовка `Authorization`.
2. Проверяет подпись access-секретом.
3. **Перечитывает пользователя из БД** и проверяет `status === ACTIVE`.
4. Если пользователь не найден или не активен — `401`.

Следствие: заблокированный/удалённый аккаунт **не сможет** доработать на ещё не
истёкшем access-токене.

## Авторизация: роли и доступ

### Глобальная защита (opt-out)

`JwtAuthGuard` — глобальный. **Любой маршрут защищён по умолчанию**; публичные
помечаются `@Public()`. Забыли декоратор — маршрут остался закрытым (fail-safe).

Для маршрутов, видимых разлогиненным, но с персональным контекстом, —
`OptionalJwtAuthGuard` (нет токена → аноним, `request.user` пуст).

### Иерархия ролей

`RolesGuard` сопоставляет роль пользователя с требованием `@Roles(...)`:

```
USER (0)  <  MODERATOR (1)  <  ADMIN (2)  <  OWNER (3)
```

Старшая роль удовлетворяет требованию младшей: `@Roles(UserRole.MODERATOR)`
пускает `MODERATOR`, `ADMIN`, `OWNER` и отклоняет `USER`. Не нужно перечислять
все вышестоящие роли на каждом маршруте.

## Прочие защиты

| Механизм | Детали |
|---|---|
| **Helmet** | базовые security-заголовки |
| **CORS** | origin из `CORS_ORIGINS`, `credentials: true` |
| **ValidationPipe** | `whitelist` + `forbidNonWhitelisted` (лишние поля → `400`) + `transform` |
| **Throttler** | глобально 120 запросов / 60 c; `register` — 5/60 c, `login` — 10/60 c |
| **Логи** | Pino редактирует заголовки `authorization` и `cookie` |
| **Ошибки** | `AllExceptionsFilter` — единый конверт, без утечки внутренних деталей |

## Чек-лист для продакшена

- [ ] `JWT_ACCESS_SECRET` и `JWT_REFRESH_SECRET` — сильные и разные
      (`openssl rand -hex 32`). В `docker-compose.yml` дефолты `*_change_me` —
      **только для локалки**.
- [ ] `CORS_ORIGINS` — реальные домены фронтенда, без `localhost`.
- [ ] `DATABASE_URL` / Redis — не дефолтные пароли.
- [ ] HTTPS перед API (токены ходят в теле/заголовках).
- [ ] Не логировать тела auth-запросов и заголовки с токенами (уже настроено).
- [ ] `.env` и секреты — вне git (см. `.gitignore`).
