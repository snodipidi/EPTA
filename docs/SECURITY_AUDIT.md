- **OAuth-привязка** — Google линкуется к существующему локальному аккаунту

## Исправления

### 1. Тайминговая enumeration при логине

**Риск.** `login()` выполнял проверку пароля только при найденном пользователе
(`user && verify(...)`). Для несуществующего email argon2 не запускался — ответ
приходил заметно быстрее, что позволяло перебором определять зарегистрированные
адреса.

**Что сделано.** `PasswordService` получил заранее посчитанный dummy-хэш
argon2id (тёплый старт в `onModuleInit` + ленивый фолбэк). `verify()` теперь
**всегда** выполняет `argon2.verify` — против реального хэша либо против dummy при
его отсутствии — и возвращает `false` для пустого хэша. В `login()` убран
короткозамыкающий вызов; verify выполняется и при отсутствии пользователя.
Сообщение об ошибке единое — `Invalid email or password`.

**Файлы:** `backend/src/auth/services/password.service.ts`,
`backend/src/auth/auth.service.ts`, `backend/src/auth/auth.service.spec.ts`.

**Побочный эффект.** OAuth-аккаунты без пароля и попытки сменить/удалить аккаунт
паролем для них ведут себя как прежде (ответ «неверный пароль»).

### 2. Валидация avatarUrl / coverUrl

**Риск.** Поля профиля `avatarUrl`/`coverUrl` валидировались лишь как строка с
ограничением длины — можно было сохранить `javascript:…`/`data:…`. Активного
sink-а в текущем UI нет (аватары рисуются SVG-иконкой), но это «спящий»
stored-XSS на случай, когда фронт начнёт рендерить эти значения как ссылку или
картинку.

**Что сделано.** `avatarUrl`, `coverUrl`, `website` (а также `avatarUrl`
группового чата) переведены на
`@IsUrl({ protocols: ['http', 'https'], require_protocol: true })`. Отклоняются
`javascript:`, `data:`, относительные и без-протокольные значения.

**Файлы:** `backend/src/profiles/dto/update-profile.dto.ts`,
`backend/src/chats/dto/create-chat.dto.ts`.

**Совместимость.** Валидация работает только на входящих запросах; чтение
существующих строк не затронуто. В сидах/дефолтах этих значений нет — миграция не
требуется.

### 3. Google OAuth — проверка email_verified и привязка

**Риск.** Стратегия брала `profile.emails[0].value` без проверки, подтверждён ли
адрес у Google, и использовала его как доверенный ключ для поиска/создания/
привязки локального аккаунта. Плюс существовала авто-привязка `googleId` к любому
локальному аккаунту по совпадению email — вплоть до неподтверждённого
(squatting) аккаунта.

**Что сделано.**
- Стратегия отклоняет вход, если Google не подтвердил адрес
  (`profile.emails[0].verified` / `profile._json.email_verified === true`).
- `loginWithGoogle()` переписан по трём ветвям:
  1. возврат пользователя по стабильному `googleId`;
  2. локальный аккаунт на этот email — **привязка `googleId` только если у него
     уже стоит `emailVerifiedAt`**; иначе `401` с просьбой войти паролем и
     подтвердить почту;
  3. иначе — создание нового, заранее подтверждённого аккаунта.
- Убрана прежняя простановка `emailVerifiedAt` при каждом входе.

**Файлы:** `backend/src/auth/strategies/google.strategy.ts`,
`backend/src/auth/auth.service.ts`.

### 4. Google OAuth — защита от login CSRF (state)

**Риск.** В OAuth-флоу отсутствовал параметр `state` — открытая дверь для
login CSRF (жертву можно завершить в сессии атакующего).

**Что сделано.** Реализован stateless double-submit без `express-session`:

- На входе (`GET /auth/google`) guard генерирует случайный `state`
  (`randomBytes(32)`, base64url), кладёт его в cookie `epta_oauth_state`
  (`httpOnly`, `SameSite=Lax`, `secure` в проде, `path=/api/auth`, TTL 10 мин) и
  передаёт passport, который подмешивает `state` в URL Google.
- На callback контроллер сверяет `state` из query с cookie (`timingSafeEqual`),
  очищает cookie (одноразовость) и при несовпадении редиректит на
  `…/auth/callback#error=invalid_state`.

Особенность `passport-oauth2`: при строковом `state` встроенный state-store
обходится, а на callback `NullStore` не вмешивается — поэтому ручная сверка
корректна и не конфликтует с библиотекой.

**Файлы:** `backend/src/auth/oauth-state.util.ts` (новый),
`backend/src/auth/guards/google-auth.guard.ts`,
`backend/src/auth/auth.controller.ts`.

### 5. Проверка содержимого загружаемых файлов

**Риск.** Тип файла определялся по клиентскому `mimetype`. Можно было загрузить
SVG/HTML/скрипт под видом `image/png`; раздаваемый по публичному URL контент
давал stored-XSS и доставку вредоносных файлов.

**Что сделано.**
- Клиентский MIME больше не влияет на решения. Реальный тип определяется по
  **magic bytes** (jpeg/png/webp/gif/mp4/webm); всё неопознанное (SVG, HTML,
  скрипты) отклоняется `400`.
- Изображения **пере-кодируются через `sharp`** — это срезает встроенные
  payload/EXIF/полиглоты и подтверждает, что байты действительно декодируются как
  изображение (плюс встроенный лимит пикселей защищает от decompression-bomb).
  Видео проверяется только по сигнатуре (без ffmpeg).
- В БД/хранилище пишется канонический MIME; ключ файла — серверный UUID с
  расширением из белого списка (без обхода пути).

**Файлы:** `backend/src/media/media.service.ts`,
`backend/src/media/media.controller.ts`, `backend/package.json` (+ `sharp`).

> Зависимость `sharp` поставляет нативный libvips. В `package-lock.json`
> присутствуют бинарники для всех платформ, включая `linuxmusl` (Alpine), поэтому
> `npm ci` в Docker-образе соберётся штатно.

### 6. Точечные rate-limit

**Риск.** Мутации были прикрыты только глобальным лимитом 120/60 c — слабо
против спама постов/комментариев/реакций, массового follow и заливки файлов.

**Что сделано.** Добавлены `@Throttle` (окно 60 с, «сбалансированный» профиль):

| Операция | Лимит / 60 c | Где |
|---|---|---|
| Загрузка файла | 20 | `media.controller.ts` |
| Создание поста / репост | 20 | `posts.controller.ts` |
| Комментарий | 40 | `posts.controller.ts` |
| Реакции (like/set/remove) | 60 | `reactions.controller.ts` (класс) |
| Follow / Unfollow | 60 | `follows.controller.ts` |
| Отправка сообщения | 60 | `chats.controller.ts` |
| Создание чата (direct/group) | 10 | `chats.controller.ts` |
| Смена пароля | 5 | `users.controller.ts` |

Глобальный лимит и инфраструктура `ThrottlerModule` не изменялись.

> Ключ лимита — IP. При деплое за обратным прокси необходимо корректно настроить
> доверие к `X-Forwarded-For`, иначе все клиенты схлопнутся в один ключ.

## Открытые находки (не входили в этот набор)

Зафиксированы аудитом, к исправлению пока не брались:

- Сброс счётчика попыток кода подтверждения при `resend-code`.
- Неатомарность `TokenService.rotate()` (внутри транзакции используется внешний
  клиент Prisma) — корректность, не безопасность.
- WS-обработчики `chat:typing` / `chat:leave` без проверки membership.
- Код подтверждения пишется в лог dev-транспортом почты (риск при
  prod-мисконфиге без `SMTP_*`).
- Enumeration через `register` (409 с именем поля `email`/`username`).
- Настройка `trust proxy` для продакшена (связано с ключом rate-limit).

**Чисто по аудиту:** SSRF, ReDoS, админ-эндпоинты (их нет), TTL access-токена
(15 мин), CORS, Prisma (параметризовано), основной IDOR.
